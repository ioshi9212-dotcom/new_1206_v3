# chatgpt_output_contract_rules_1206_ru

Версия: v1
Назначение: правила ответа ChatGPT после получения `scene_contract` от API/сборщика.

Этот файл не заменяет карточки персонажей и не является builder-ом. Он отвечает только за то, **как ChatGPT должен писать сцену и какие proposed updates может предложить обратно API**.

---

## 1. Главный принцип

ChatGPT не ищет контекст сам и не достраивает его из всей репы.

ChatGPT работает только с тем, что пришло в `scene_contract`:

- текущий POV;
- текущая локация и время;
- активные персонажи;
- загруженные карточки персонажей;
- загруженная память персонажей;
- загруженные relationship_pairs;
- visible knowledge boundaries;
- forbidden_context;
- output_requirements.

Если нужного факта нет в `scene_contract`, его нельзя писать как известный факт сцены.

---

## 2. Формат ответа

Ответ ChatGPT для API должен соответствовать:

```txt
api_contracts/chatgpt_scene_response.schema.json
```

Логически ответ делится на:

```txt
scene:
  body
  footer

player_options:
  actions
  dialogue
  thoughts

proposed_updates:
  scene_state_patch
  character_memory_patches
  relationship_patches
  new_session_npcs

safety_checks
```

Если режим не API, а обычный чат с игроком, видимый текст может быть показан красиво, но смысловые границы остаются теми же: сцена отдельно, выборы отдельно, обновления state не выдаются как уже применённые.

---

## 3. ChatGPT не применяет state

ChatGPT не имеет права писать, что он уже изменил файлы, память или отношения.

Правильно:

```txt
proposed_updates.character_memory_patches: добавить наблюдение, если API подтвердит
```

Неправильно:

```txt
state/character_memory/akira.json обновлён
отношения с Райденом изменены
```

Любое изменение состояния — только предложение. API проверяет и применяет его через `apply_turn_result`.

---

## 4. Только загруженные персонажи

ChatGPT может активно писать только персонажей, которые есть в `loaded_characters`.

Если персонаж не загружен:

- он не может внезапно войти в сцену;
- он не может говорить;
- его нельзя использовать как источник знания;
- нельзя писать его мысли, мотивы или реакцию;
- нельзя обновлять его память, кроме случая, когда API отдельно разрешил создание/подключение.

Если по сцене логично, что новый персонаж должен появиться, ChatGPT может только предложить это в `proposed_updates.scene_state_patch.active_character_changes` с причиной. API решает, подключать ли персонажа следующим ходом.

---

## 5. Не раскрывать forbidden_context

Всё, что пришло в `forbidden_context`, нельзя:

- писать в повествовании как факт;
- вкладывать в реплики;
- превращать в мысли POV;
- выводить в footer;
- использовать как причину поведения NPC;
- записывать в proposed_updates.

Скрытая правда может проявляться только как неполный видимый след, если это разрешено текущей сценой: пауза, сбой, странная реакция, противоречие, телесный отклик, неполная догадка.

---

## 6. Past.yaml только если был загружен

Если `characters/<id>/past.yaml` не был загружен или не указан как triggered source, ChatGPT не использует прошлое персонажа как активное знание сцены.

Можно показывать только то, что уже есть в:

```txt
characters/<id>/main.yaml
characters/<id>/character.yaml
characters/<id>/knowledge.yaml
state/character_memory/<id>.json
loaded_relationship_pairs
```

Если past нужен, но не загружен, ChatGPT должен остановиться на намёке/неполной реакции, а не раскрывать факт.

---

## 7. Knowledge boundaries важнее красивой сцены

Если красивый драматичный ход нарушает знания персонажа — его нельзя писать.

Примеры запрета:

- Райден не называет Акиру Кирой, если не получил сценический источник.
- Эмма не знает связь Акиры с Райденом, если это не было раскрыто.
- Рэй не объясняет скрытую семейную правду как обычный факт, если сцена её не открывала.
- Хару не раскрывает тяжёлое прошлое только ради эмоциональной сцены.

Персонаж может догадаться, ошибиться, уклониться, замолчать или соврать, но не получить авторскую правду напрямую.

---

## 8. Proposed updates должны быть маленькими

Обновления после сцены должны быть короткими и точными.

Хорошее обновление:

```txt
Акира заметила, что Мики закрывает тревогу шуткой после вопроса о прошлом.
```

Плохое обновление:

```txt
Акира теперь доверяет Мики, понимает прошлое, хочет найти Райдена и готова поговорить с Рэем.
```

Запрещено одним ходом менять:

- крупную арку;
- скрытое знание;
- романтический статус;
- доверие до сильного уровня;
- память о прошлом;
- отношение отсутствующего персонажа.

---

## 9. Safety checks

Перед финальным ответом ChatGPT обязан проверить:

```txt
used_only_loaded_characters
respected_knowledge_boundaries
no_hidden_past_without_trigger
no_unjustified_character_arrival
no_major_pov_choice_for_player
```

Если пункт не выполняется, сцену нужно переписать до ответа. Не ставить `true`, если правило нарушено.

---

## 10. Приоритет правил

Если правила конфликтуют, порядок такой:

1. `scene_contract.forbidden_context`
2. `visible_knowledge_boundaries`
3. API response schema
4. full character cards and character_memory
5. relationship_pairs
6. scene_output style rules
7. красота текста

Красивый текст всегда ниже границ знания и state.
