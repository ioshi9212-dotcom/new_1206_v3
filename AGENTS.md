# AGENTS.md — read before changing this repository

Перед любыми изменениями сначала прочитать этот файл.

## Главная цель

Не плодить новые lock/patch-слои поверх старых. Сначала найти файл-владелец проблемы, потом предлагать точечное изменение.

## Что это за ZIP / слой

Этот пакет может быть content/state/style layer, а не полным Railway app. Не считать его автоматически полноценной рабочей репой, если здесь нет `app/`, `requirements.txt`, `Procfile`.

## Правило безопасной работы

1. Не заменять `characters/` целиком, если пользователь уже чистил персонажей вручную.
2. Не перетирать ручные правки полным ZIP без явного разрешения.
3. Сначала дать diff/план: какие файлы трогаются и зачем.
4. Если нужно добавить правило — проверить, нет ли уже существующего владельца этого правила.
5. Не создавать новый `*_runtime_patch.py`, если можно исправить существующий builder/state/prompt файл.
6. Не возвращать старые пути динамической памяти. Текущий стандарт: `state/character_memory/<id>.json`.

## Архитектура

- `characters/<id>/` — главный источник статичных карточек персонажей.
- `state/character_memory/` — динамическая память персонажей.
- `state/relationship_pairs/` — отношения парами.
- `canon_lore/` — общий лор.
- `calendar/` — сюжетный календарь.
- `gpt/scene_output/` — стиль, POV, footer, качество сцены.
- `state/context_loading/` — правила загрузки контекста.
- `app/` — код Railway, если присутствует.

## Критические запреты

- Не грузить Акиру автоматически во все POV.
- Не грузить все пары Акиры по умолчанию.
- Не делать hidden lore знанием NPC.
- Не смешивать `state/character_memory` со старым dynamic-state слоем.
- Не заменять очищенные пользователем character cards старыми версиями.

## Перед выдачей ZIP

Явно написать, что это:
- полный репозиторий,
- patch-only,
- content-layer,
- docs-only,
- или code patch.

Если это не полный Railway app — так и написать.

## Источник поведения персонажей

Если персонаж нужен в сцене, сборщик должен брать данные напрямую из полных карточек и state, без отдельного короткого слоя-пересказа.

Порядок источников:

1. `characters/<id>/main.yaml` — паспорт, роль, внешность;
2. `characters/<id>/character.yaml` — характер, голос, поведение, привычки, бой, энергия;
3. `characters/<id>/knowledge.yaml` — постоянные знания/незнания;
4. `state/character_memory/<id>.json` — динамическая память;
5. `state/relationship_pairs/<pair>.json` — только если пара релевантна;
6. `characters/<id>/past.yaml` — только по триггеру прошлого.

Не создавать отдельные короткие summary-файлы вместо полных карточек. Они уплощают персонажей и ломают поведение.
