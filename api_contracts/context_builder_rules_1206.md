# 1206 — правила API-сборщика scene_contract

Этот файл описывает, как API/сборщик должен готовить контекст для ChatGPT.
Цель: убрать старую проблему 1206 v2 и Академии-приквела, где данные были разбросаны, а персонажи часто собирались из коротких выжимок или fallback-логики.

---

## 1. Главный принцип

Поведение персонажа строится только из полных карточек и актуальной памяти:

```txt
characters/<id>/main.yaml
characters/<id>/character.yaml
characters/<id>/knowledge.yaml
state/character_memory/<id>.json
```

`characters/<id>/past.yaml` грузится только условно.

Отдельные короткие summary-файлы персонажей не создавать и не использовать как источник поведения. Они уплощают характер и ломают сцены.

---

## 2. Что грузить для активного персонажа

Если персонаж активен в сцене, физически присутствует, говорит, к нему обращаются или scene_goal требует его участия, сборщик грузит:

```txt
characters/<id>/main.yaml
characters/<id>/character.yaml
characters/<id>/knowledge.yaml
state/character_memory/<id>.json
```

`main.yaml` даёт паспорт/роль/внешность.
`character.yaml` даёт поведение, речь, реакции, слабости, бой.
`knowledge.yaml` даёт стартовые знания и незнания.
`character_memory` даёт изменяемую память и актуальные следы сцен.

Нельзя заменять это одной короткой строкой вроде “холодный командир” или “шумный тёплый парень”.

---

## 3. Условная загрузка past.yaml

`characters/<id>/past.yaml` не грузится в каждый ход.

Грузить только если есть триггер:

```txt
флэшбек
разговор о прошлом
scene_goal прямо требует прошлое
персонаж вспоминает прошлое
другой персонаж произносит факт из прошлого
текущая дата/событие календаря требует прошлого
```

Если триггера нет, скрытое прошлое не попадает в `scene_contract`.
ChatGPT не может использовать незагруженное прошлое как мотивацию, реплику или знание персонажа.

---

## 4. Акира не грузится автоматически

Запрещён старый fallback-стиль:

```txt
если scene_ids пустые → вставить Акиру
если Акиры нет → добавить Акиру
```

Правило v3:

```txt
если POV не Акира
и Акира физически отсутствует
и её не упомянули с прямой сценической важностью
и scene_goal не про неё
и нет нужной relationship_pair с ней
→ Акиру не грузить
```

Акира — персонаж, а не глобальный центр всех сцен.

---

## 5. Relationship pairs

`state/relationship_pairs/<a>__<b>.json` грузить только если пара реально нужна:

```txt
оба персонажа присутствуют
один говорит о другом
текущий конфликт/действие связано с этой парой
recent memory требует обновить отношения
scene_goal прямо требует пару
```

Не грузить все пары Акиры подряд.
Не грузить все отношения базы одним дампом.

---

## 6. Knowledge boundaries

Для каждого загруженного персонажа сборщик должен дать ChatGPT границы знания:

```txt
known: что персонаж точно знает
unknown: чего персонаж не знает
must_not_assume: что нельзя додумывать без источника
```

Это нужно, чтобы не повторялись ошибки:

```txt
NPC называет имя без источника
Райден знает Акиру раньше времени
персонаж знает hidden past без сцены
постовые знают личность гостя без документа
```

Если источник знания не найден в `knowledge.yaml`, `character_memory`, сцене или загруженной паре отношений — персонаж этого не знает.

---

## 7. Forbidden context

Сборщик должен явно отдавать список `forbidden_context`.

Примеры:

```txt
raiden_memory_before_trigger
akira_hidden_kairos_origin
lost_child_information
samuel_lab_details_without_source
ray_family_role_visible_to_akira_without_scene
```

ChatGPT не должен использовать эти факты в сцене, если они не были загружены по триггеру.

---

## 8. Новые NPC

Фоновые NPC не сохраняются автоматически.

Сохранять NPC только если:

```txt
он получил имя
он повторяется
у него есть функция в сцене
он стал свидетелем важного факта
он влияет на сюжет/допуск/опасность/отношения
```

Обычный прохожий, медик без имени, постовой без роли — не отдельная карточка.

---

## 9. ChatGPT не пишет state напрямую

ChatGPT возвращает только предложения:

```txt
proposed_updates.scene_state_patch
proposed_updates.character_memory_patches
proposed_updates.relationship_patches
```

API применяет их только после проверки через `apply_turn_result.schema.json`.

Если ChatGPT предложил записать факт, который персонаж не мог узнать, патч отклоняется.

---

## 10. Минимальная проверка перед отправкой scene_contract

Перед вызовом ChatGPT сборщик проверяет:

```txt
1. У каждого active_character есть main/character/knowledge/memory.
2. past.yaml загружен только при явном триггере.
3. relationship_pairs выбраны точечно.
4. Акира не добавлена fallback-ом.
5. Нет коротких summary-источников как основы поведения.
6. visible_knowledge_boundaries заполнены.
7. forbidden_context заполнен для hidden/опасных фактов.
8. last_user_turn передан без переписывания смысла.
```

Если проверка не проходит — scene_contract не отправлять в ChatGPT, а возвращать diagnostics/repair.
